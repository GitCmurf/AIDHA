# Critical Evaluation and Teaching Points for DocOps Drafts

First, let me reiterate: this is an excellent, C.V.-worthy piece of systems design. My critique is aimed at making it "battle-hardened" and increasing its chances of adoption.

The entire philosophy of my feedback is: Reduce Friction, Remove Redundancy, and Rely on Tooling (eventually).

Key Teaching Points

1. The Enemy: Friction

The single greatest threat to your system is not a bad document; it's a good document that's out of date. Why do docs get out of date? Friction.

If a developer fixing a bug has to: 1) Find the doc, 2) Understand the ID system, 3) Update the YAML, 4) Update the redundant visible block, 5) Update the redundant version history table... they will simply not do it.

Your Goal: Make updating a document almost as easy as writing a commit message.

2. DRY (Don't Repeat Yourself) & SSOT (Single Source of Truth)

This is a core programming principle that applies 100% to documentation:

Your YAML block is a "Single Source of Truth" (SSOT) for machines.

Your Git history is the SSOT for changes over time.

Any time you manually duplicate this information (in a visible metadata block or a version table), you are violating DRY. This manual sync will fail and the data will drift.

Solution: Remove the manual redundancy. Rely on the SSOT.

3. Let Tools Do the Work

Your Constitution is a "spec" for a documentation system. The best specs are enforced by tools.

Problem: Your Article I has redundant YAML and visible blocks.

Long-Term Solution: A static site generator (like MkDocs, Docusaurus, Astro) can read the YAML front matter (the SSOT) and automatically generate the nice-looking visible metadata block on the webpage.

My Proposal: Since we can't assume that tooling exists yet, my redraft will focus on removing the most painful manual duplication (the version table) immediately.

Article-by-Article Critique (The Constitution)

Here is a breakdown of my thoughts on your DOCOPS_CONSTITUTION_v1.2.

Article I — Canonical Metadata:

Critique: As discussed, the YAML front matter and the visible block are redundant. This is a friction point.

Recommendation: I've left this as-is in the redraft because the visible block is user-friendly. However, I've added a teaching point in this critique: the long-term goal should be to auto-generate the visible block from the YAML.

Article II — Document Identification:

Critique: II.3 Uniqueness: Document IDs must be unique across the organisation. This is a very hard requirement. It implies a central, global registry that all teams must query before creating a doc. This is high-friction.

Recommendation: You already solved this! Repo prefixes provide namespace isolation. This is the correct answer. I've re-worded this article to make it clear that uniqueness is only required within the repo's prefix. OZYS-INGEST-ADR-001 and OZYS-EDITOR-ADR-001 are both fine.

Article IV — Versioning:

Critique: Changes to metadata fields do count as content changes. This is a bit too strict. If I just fix a typo in the owner field, does that really warrant a v1.2 -> v1.3 bump?

Recommendation: I've added a clarifying clause to soften this. Substantive metadata changes (like status: Draft -> Approved) should increment the version. Trivial typo-fixes should not.

This is the single biggest improvement you can make to your system. It removes a massive, redundant burden from every single document change.

Recommendation: I've softened the language to ...must use the following structure... Directories may be omitted if no documents of that type exist. This maintains the consistent numbering (convention) without forcing empty directories (configuration).

Critique: This is the "Holy Grail" article. It's perfect. It's the entire point of DocOps.

Recommendation: No changes. I love it. I've maybe strengthened the wording slightly.

# DOCOPS CONSTITUTION v1.3

Organisation-wide governance for technical documentation *

Document ID: ORG-DOCOPS-CONSTITUTION Owner: DocOps Working Group Approvers: — Status: Approved Version: 1.3 Last Updated: YYYY-MM-DD DocOps Version: 1.3

Owner: DocOps Working Group

Approvers: —
Status: Approved

Version: 1.3

Last Updated: <Date>

Preamble

This Constitution defines the mandatory, organisation-wide rules governing the creation, structure, identification, versioning, and maintenance of technical documents. It ensures consistent, discoverable, and interoperable documentation across all repositories. Repository-level standards may extend this Constitution but may not contradict it.

Article I — Canonical Metadata

Every governed document must begin with a YAML front matter block and a human-visible metadata block. The YAML block is the canonical "Single Source of Truth."

I.1 Required YAML Front Matter

Every governed document must begin with a YAML block containing:

document_id: <ID>
owner: <Person or Role>
approvers: <List or "—">
status: <Draft|In Review|Approved|Published|Superseded>
last_updated: <YYYY-MM-DD>
version: <MAJOR.MINOR>
type: <prd|adr|fdd|runbook|design|spec|decision|governance|strategy|planning|index>
docops_version: <Version of this Constitution>


Additional metadata keys are allowed for domain- or automation-specific purposes.

I.2 Visible Metadata Block

Below the YAML block, a human-visible block must mirror the key metadata:

Document ID: <ID>

Owner: <Person/Role>

Approvers: <List or "—">

Status: <Status>

Version: <Version>

Last Updated: <Date>

Note: In the absence of tooling, this block must be manually kept in sync with the YAML front matter.

I.3 Sidecar Metadata

Files that cannot contain front matter (e.g., diagrams, binary specs) must have a sidecar:

<filename>.meta.yaml

containing the canonical metadata. Ephemeral build outputs are exempt.

Article II — Document Identification

II.1 Format

Document IDs must follow:

<REPO_PREFIX>-<AREA>-<TYPE>[-<SEQ>]

<REPO_PREFIX>: Repository prefix (e.g., OZYS, INFRA)

<AREA>: Functional domain (e.g., INGEST, AUTH)

<TYPE>: Document type (e.g., ADR, PRD)

<SEQ>: Optional three-digit sequence (e.g., 001)

II.2 Sequence Rules

Sequence numbers are mandatory for:

Architecture Decision Records (ADRs)

Feature Design Documents (FDDs)

Any document type where multiple instances are expected (e.g., RUNBOOK).

II.3 Uniqueness

Document IDs must be unique. The <REPO_PREFIX> and <AREA> fields provide namespace isolation. Uniqueness is only guaranteed within a given namespace.

Article III — Status Vocabulary

Permitted statuses:

Draft: Work in progress, not for reliance.

In Review: Seeking approval from peers or approvers.

Approved: Content is agreed upon, but may not be implemented.

Published: Implemented, "live," or the official current standard.

Superseded: No longer in use, replaced by a new document.

Superseded documents must state:
> **This document is superseded by:** <Document ID> v<Version>

Superseded documents should be moved to the 99-archive/ directory.

Article IV — Versioning

IV.1 Format

Semantic versioning is used: MAJOR.MINOR

IV.2 Rules

Increment MAJOR when the scope, structure, or fundamental assertions of the document change. A new set of approvers is required.

Increment MINOR for additive revisions, clarifications, or corrections that do not change the core assertions.

Metadata Changes: Substantive metadata changes (e.g., changing owner, approvers, or status from 'Draft' to 'Approved') require a MINOR version increment. Trivial corrections (e.g., fixing a spelling error in an author's name) do not.

Article V — Change Control

The source control system (Git) is the canonical version history for all documents.

A manually maintained "Version History" table is explicitly forbidden, as it is redundant and error-prone.

All changes to documents must be made via Pull Requests.

The version and last_updated metadata fields in the YAML front matter must be updated as part of the Pull Request that modifies the document.

Article VI — Repository Document Structure

Each repository must use the following structure under a root docs/ folder. Directories may be omitted if no documents of that type exist.

docs/
  00-governance/  # Policies, this standards file, CONTRIBUTING.md
  01-indices/     # Glossaries, Tables of Contents
  02-strategy/    # Vision, Business Case, Problem Definition
  05-planning/    # Roadmaps, Schedules
  10-prd/         # Product Requirements Documents
  20-specs/       # API specifications, Data schemas, Contracts
  30-design/      # System diagrams, Architecture overviews
  40-decisions/   # Technical decisions, Tech stack
  45-adr/         # Architecture Decision Records
  50-fdd/         # Feature Design Documents
  55-testing/     # Test strategies, Test plans
  60-runbooks/    # Operational runbooks, Deployment guides
  70-devex/       # Local setup guides, Dev workflows
  99-archive/     # Superseded documents


Article VII — Linking and References

Documents must:

Reference other documents by their full Document ID in prose.

Use relative markdown links for navigation within the repository.

Maintain stable references using Document IDs, not titles (which can change).

Article VIII — Coupling Between Code and Documentation

This is the core principle of DocOps.

Any Pull Request that changes system behaviour, API boundaries, user impact, constraints, or operational procedures must include corresponding updates to the relevant documentation in the same Pull Request.

Article IX — Orthography and Filenames

Filenames and identifiers must use ASCII (32–126) only.

American spelling (en-US) is required for filenames and identifiers.

Document content may use either British (en-GB) or American (en-US) spelling, but must be internally consistent.

Repository-level spelling exceptions may be declared via dictionary files.

Article X — Local Extensions

Repositories may define additional local rules in:
docs/00-governance/LOCAL_STANDARDS.md

Local rules must not contradict this Constitution.

Article XI — Compatibility

(No changes from your v1.2)

Article XII — Amendment Procedure

(No changes from your v1.2)

End of DocOps Constitution v1.3

adr (Architecture Decision Records)

fdd (Feature Design Documents)
